语言名称：Lency
文件后缀：`.lcy`

在设计阶段，应先定义清晰且完整的长期目标，明确该语言的差异化特性，而不是对现有语言进行简单照搬。
Lency 不以实现 C/C++/Rust 这类底层语言为目标；其方向更接近 Go，即在简洁基础上形成自身特色。

必须在上层设计阶段就构建完整的结果形态与内容边界。否则在后期扩展与完善中，极易出现大量 `if-else` 式补丁化实现。

Lency 的核心构想：

- 认可 Python 的简洁性；但层级规范不应依赖“强制缩进语义”。缩进应作为推荐格式，而非强制规则。
- 认可“C 系”书写规范（此处“C 系”指 C/Java/C++ 一类风格，不含 Rust 风格）。
- 语言必须是静态类型；语义结果应在编译期确定。可以支持多态，但不能放任内存泄漏与空指针问题。
- 语言与代码书写都不应走向 Java 式冗长。
- 诊断必须清晰：应能准确定位问题来源，接近 Rust 的错误可读性，而不是产生大量冗长堆栈噪音。
- 开发过程必须稳扎稳打；宁可功能推进更慢，也要保证每一步实现正确。

---

Lency 的语言哲学是：简洁、规范、清晰，并附加安全性约束。后续所有语言特性演进都不得违背该哲学。
此外，语言设计应尽量避免运行时错误。

语言设计思想参考 Go 的“组合优于继承”，避免 Java 式大量冗长语法，持续保证语言简洁性。

该语言最终必须实现自举：即使用 Lency 实现 Lency 编译器。

凡是需要读者模拟编译器推理过程才能理解的地方，均视为“黑魔法”。
Lency 中禁止黑魔法，尤其禁止泛型推断黑魔法。
